Operating truths

The simple rules underneath the system

One place for the truth

Live execution belongs in the checklist. Longer follow-up belongs in Task Center. Coverage belongs in schedule tools.

Every exception needs a reason

Skipping and reporting are allowed, but silence is not. The reason becomes audit trail and handover cue.

Handover is part of the work

The next operator must inherit decisions, blockers, risks, and unfinished follow-up in writing.

Close from evidence

A clean close and a close with exceptions mean different things. SentinelOps preserves both clearly.

Smart Subitem modal

Work subitems in order

Subitems are the guided path inside a detailed item.

Use Start Working, Mark Complete, Skip, or Report Issue
  1. Use Next or Continue to move through actionable subitems.
  2. Mark Complete only when the subitem is done.
  3. Use Skip or Report Issue with a clear reason when normal completion is not possible.
If any subitem is reported failed, SentinelOps requires a final verdict before the main item can be completed.

Complete clean work cleanly

Use Mark Complete when the item or subitem has been performed and no exception remains.

Click Mark Complete
  1. Confirm the action was actually performed.
  2. Add concise completion evidence when needed.
  3. For items with subitems, complete all subitems before completing the main item.

09
Item Actions or subitem reason modal

Skip only when it is valid to skip

Skip when the step is not applicable, blocked by a valid condition, or deliberately deferred by the operating process.

Click Skip Item or Skip
  1. Choose Skip only after deciding the step should not be completed now.
  2. Write the reason in plain language.
  3. If skipped work still matters later, add handover or create a task.
Skipping is not the same as resolving. It is an exception record.

10
Item Actions or subitem reason modal

Report issues instead of hiding failure

Report Issue when execution failed, evidence shows a real problem, the action is unsafe, or escalation is needed.

Click Report Issue
  1. State what failed or what symptom was observed.
  2. Include evidence, system names, timestamps, or impact.
  3. After investigation, use Resolve & Complete or add final verdict when ready.
Do not convert a failure into a skip just to finish faster.

Finalize the checklist honestly

Managers and admins complete the checklist from the header once work is actioned.

Click Complete Checklist
  1. Confirm every item is completed, skipped, failed, or otherwise actioned.
  2. Review exception count in the dialog.
  3. Confirm Complete, then download the checklist PDF if a record is needed.
If exceptions exist, the checklist should close with exceptions. That is transparency.
Decision guide

When to complete, skip, report, hand over, or finalize

Start Working

Use when

You are actively taking responsibility for an item now.

Do this

Open item actions, choose Start Working, and add a short note if context matters.

Evidence

The item moves out of pending state and participant engagement is visible.

Mark Complete

Use when

The work was performed and no exception remains.

Do this

Choose Mark Complete and add concise notes when evidence or timing matters.

Evidence

The item or subitem shows Completed with completion metadata.

Skip Item

Use when

The step is not applicable, blocked by a valid condition, or deliberately deferred.

Do this

Write the reason, then hand over or create a task if follow-up still matters.

Evidence

The reason remains visible and the checklist can close with exceptions if needed.

Report Issue

Use when

Something failed, is unsafe, is unclear, or needs escalation.

Do this

Describe symptom, impact, evidence, and immediate next action.

Evidence

The issue is retained as an exception and can require final verdict.

Add Final Verdict

Use when

An item had skipped or failed evidence but is now ready to close.

Do this

Summarize the outcome, what changed, and why the item can be considered complete.

Evidence

The verdict appears beside the completed item for audit and handover review.

Add Handover Note

Use when

The next shift needs context, warning, dependency, evidence, or a decision trail.

Do this

Choose priority, write operational facts, and focus on what happens next.

Evidence

Incoming and outgoing notes appear with priority and author.

Workspace map

Use the right room for the right kind of work

Dashboard

Question: What is happening right now?

Read operational state, Shift Radar, command threads, and handover summary.

Checklist Timeline

Question: Which run, date, shift, or pattern do I need to inspect?

Choose All Time, Week, Specific Day, or Date Range, then filter by shift and status.

Live Checklist

Question: What work must be executed and recorded now?

Join, work items truthfully, keep handover notes and team presence current.

Task Center

Question: What needs accountable follow-up beyond this shift?

Use personal, assigned, team, overdue, and completed lanes for durable ownership.

Handover Notes

Question: What must the next operator know?

Acknowledge incoming notes, add outgoing notes with priority, resolve or promote notes as needed.

Templates

Question: How do we make repeated work easier next time?

Order items the way work naturally happens and publish only after reviewing operator flow.

Schedule and Team Management

Question: Who covers what, and when?

Choose planning horizon, read coverage, use patterns for recurring structure.

Network Sentinel and TrustLink

Question: What evidence proves service or pipeline state?

Read Signal, Timeline, Evidence, and pipeline stage state before taking control action.
